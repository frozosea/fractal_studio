# M3 Asset And Media Library — Implementation Gap Audit

**Scope.** This audit compares M3 in
[`platform-backend-spec.md`](../platform-backend-spec.md) with the implementation
present on branch `platform_backend` (audit snapshot: `abc8461`). It covers M3
HTTP/API boundaries, ingestion, storage, derivatives, deletion and M3-owned
outbox work. It does not treat planned M5/T11 functionality as an M3 defect;
those dependencies are called out separately.

## What Is Already Aligned

- M3 has owner-scoped library routes, cursor pagination, hide/restore, soft
  delete and master-download issuance.
- Verified Compute ingestion creates an asset in `processing`, uploads the
  selected master, finalises the asset as `ready`, and emits
  `media.create_derivatives.v1`.
- Master and derivatives use separate private/public key prefixes. Browser DTOs
  omit object keys, Compute artifact URLs and run IDs.
- Image and video derivative generation exists. The database constrains one
  `asset_file` per `(asset_id, purpose)`.
- Soft delete checks non-archived listings and retains master bytes when an
  `order_item` or active entitlement references the asset.

## Confirmed Gaps

| Priority | Gap | Evidence | Impact | Required closure |
|---|---|---|---|---|
| Blocker for M4 | **No `AssetReader` M3 port or implementation.** M3 specification requires `findOwnedAsset`, `findPublicPreview` and `findPublishableAsset`; M4 must use this narrow port instead of an M3 repository. | `app/assets/ports.py` defines only `AssetIngestionPort` and `EntitlementReader`. `app/marketplace/ports.py` is only a placeholder. Task 09 explicitly depends on these three methods. | Catalogue/listing code cannot validate ownership/readiness or obtain safe preview metadata without breaking module boundaries. | Add M3 `AssetReader` protocol, safe DTOs and repository-backed adapter. Do not return master keys or generic asset rows. |
| High | **Default private-library list includes hidden assets.** Spec says `hidden` removes an asset from creator library UI; restore must be explicit. | `app/assets/repository.py:list_owned()` filters only `status <> 'deleted'` when no status filter is supplied. It does not exclude `visibility = 'hidden'`. | `GET /v1/me/assets` contradicts hide semantics and exposes hidden assets in normal UI. | Default query must filter `visibility = 'private'`; a deliberate `visibility=hidden` filter may expose hidden assets for restore UI. Add route/integration tests. |
| High | **No safe public-preview delivery contract.** M3 produces files under `public/previews/...`, but exposes neither a `findPublicPreview` result nor preview URL/URL policy. | `AssetFileView` intentionally omits `object_key`; `ObjectStorage` only issues signed GET URLs for masters; no M3 public preview adapter exists. | M4 cannot serve catalogue cards/details without leaking storage implementation or guessing bucket/CDN behaviour. | Define a safe preview DTO (`thumbnailUrl`, `watermarkedPreviewUrl`, `videoPosterUrl`, media metadata). Choose and document one delivery model: CDN/public read policy or short-lived preview URLs. Keep master URL issuance in M3 authorization path. |
| High | **Derivative upload is not failure-compensated.** S3 uploads happen before database records; a later upload or DB failure leaves untracked preview bytes. | `MediaWorker._upload_and_record()` uploads each derivative, then opens one DB transaction. It never creates `storage_cleanup_tasks` for already uploaded derivative keys on failure. | Orphaned public objects accumulate; retries/dead letters have no reliable cleanup path. | Track uploaded keys and append M3 cleanup tasks/outbox work on failure. Make cleanup idempotent. Add tests for first-upload, second-upload and DB-commit failures. |
| High | **Media worker may record a derivative after concurrent delete/state change.** | `repository.add_derivative()` runs `SELECT ... status = 'ready' FOR UPDATE` but ignores whether a row was returned, then always executes the `INSERT`. | A derivative row/object can be attached after asset becomes `deleted` or otherwise ceases to be ready. | Require locked ready row (`one_or_none`/rowcount check) and abort + compensate uploaded object when absent. Define race behaviour with soft delete. |
| High | **Scheduled expiry cleanup is absent.** M3/M7 specify `cleanup.expired.v1` and periodic eligibility scanning owned by M3 predicates. | `build_render_handler_registry()` registers only `asset.cleanup_orphan.v1`; `OutboxWorker` has no M3 due-work reader. `AssetCleanupService` only processes a supplied orphan task. | Retention expiry, deferred physical deletion and abandoned objects have no lifecycle implementation. | Add M3 cleanup eligibility reader, `cleanup.expired.v1` handler and retry/dead-letter behaviour. Keep domain transition in M3 and scheduling in M7. |
| High | **Mesh assets have no path to required marketplace preview.** M3 specification says mesh can be published only after a PNG preview exists. | `MediaWorker._expected_purposes()` returns an empty set for every non-image/non-video asset; `_generate()` also returns an empty list. | GLB/STL assets can become `ready` but never meet M4 publish-preview requirement. | MVP choice: accept a verified PNG preview artifact from Compute during ingestion, or add a mesh-preview renderer. M4 must reject meshes without preview. |
| Medium | **No media failure state, audit event or dead-letter remediation.** | `media.create_derivatives.v1` has no dead-letter handler in `build_render_handler_registry()`. Asset remains `ready` if media conversion repeatedly fails. | Creator sees a finished library item but listing remains impossible; operators have no durable reason/action trail. | Record derivative failure reason/status or audit event; register a dead-letter handler; expose a retry/rebuild action only if product needs it. |
| Medium | **Media subprocess/resource limits are incomplete.** | `MediaWorker._run_ffmpeg()` has no timeout or process-size limit. Temporary directory has no configured quota. Pillow has a pixel cap, but decompression-bomb warnings are not promoted to errors. | Malformed/expensive masters can exhaust worker CPU, disk or process slots. | Add configurable ffmpeg timeout, derivative size/duration limits, temporary-storage quota and hard image-bomb handling. Map failures to a stable retry/non-retry error class. |
| Medium | **Storage security/delivery policy is implicit.** M3 spec names S3/MinIO encryption configuration and separates private masters from public previews. | `ObjectStorage` sends no encryption/header policy and code/docs do not define bucket policy, CDN origin or preview cache policy. | Deployments can accidentally make masters public or leave previews inaccessible. | Add environment/config contract and deployment policy: private master bucket/prefix, required SSE/KMS where applicable, preview read model, CORS and cache headers. Verify it in integration tests. |
| Medium | **Implementation structure differs from M3 diagram.** Diagram presents one `AssetService`; code splits `AssetIngestionService` and `AssetLibraryService`, and media worker directly uses repository/storage. | `app/assets/service.py`, `app/assets/media_worker.py`. | Not a runtime bug, but diagram no longer tells implementers where M3 responsibilities live. | Either update M3 module diagram/source layout to show the split, or introduce a small façade. Preserve current bounded responsibilities. |

## Planned Dependency, Not a T08 Defect

| Item | Current state | Closure point |
|---|---|---|
| Buyer master downloads | `AssetLibraryService` accepts `EntitlementReader`, but defaults to `DenyEntitlementReader`; owner downloads work while purchaser downloads always deny. | Wire an M5 implementation in T11. Add integration test: active entitlement permits download; revoked/missing entitlement denies. |

## Test Coverage Gaps

Current unit tests cover safe asset DTO filtering and basic image derivative output.
The opt-in E2E test covers image rendering, owner download issuance and a basic
hide flow. Missing automated coverage:

- default library excludes hidden assets; explicit restore path returns them;
- soft delete with each retention condition (`order_item`, active entitlement,
  neither) and eventual cleanup;
- derivative failure compensation and concurrent soft-delete race;
- video derivative files and media metadata;
- mesh preview policy;
- public preview delivery without key leakage;
- signed master URL fetch, TTL and purchaser entitlement;
- `cleanup.expired.v1` scheduling, retry and dead-letter behaviour.

## Suggested Closure Order

1. Add `AssetReader` and public-preview contract; this unblocks M4 without
   leaking storage details.
2. Fix default hidden filtering and derivative state/compensation race.
3. Select mesh-preview MVP approach and enforce it in M3/M4 publishability.
4. Implement M3 scheduled cleanup and media dead-letter handling.
5. Wire M5 entitlement reader and add integration coverage for retention and
   purchaser downloads.
